Flujo de trabajo del equipo.

1.- Objetivo General del Flujo de Trabajo:

    El objetivo de este documento es definir, de manera clara y operativa, cómo colaboran los Equipos de Data Science, Backend, Frontend, UX/UI y Diseño durante el desarrollo del Proyecto.

    Aquí se establece:
    -   Qué tareas realiza cada área.
    -   En qué carpetas del proyecto deben trabajar:
    -   El ordne en el que se entregan los productos entre equipos.
    -   Reglas mínimas de colaboracion, control de versiones y buenas prácticas.
    -   Un flujo visual del proceso completo para facilitar la comprensión.


2.- Vista General del Flujo de Trabajo.

    El desarrollo del proyecto sigue un proces secuencial y colaborativo entre los equipos:

    1.- Diseño produce identidad visual y assets.
    2.- UX/UI define la experiencia y prototivos navegables.
    3.- Data Science procesa los datos, genera modelos, y los entrega al backend.
    4.- Backend implementa servicios API que ejecutan el modelo.
    5.- Frontend consume la API y construye la interfaz funcional.
    6.- Integración final revisa coherencia, funcionalidades y UX.

    Diagrama - Flujo General de Equipos.
        A[Diseño] --> B[UX/UI]
        B --> C[Data Science]
        C --> D[Backend]
        D --> E[Frontend]
        E --> F[Integración Final]

3.- Flujo de trabajo del equipo de diseño.

    Carpetas utilizadas:
        /design
            ├─ assets/
            ├─ branding/
            └─ logo/

    Actividades principales:
    -   Crear la identidad gráfica del proyecto (logos, tipografías, colores).
    -   Diseñar assets reutilizables por el equipo de frontend.
    -   Mantener actualizados los elementos visuales oficiales.
    -   Generar especificaciones para la integración con el equipo UX / UI.

    Diagrama - Flujo del equipo de diseño:
        A[Creación de Branding] --> B[Producción de Assets]
        B --> C[Exportación de Archivos]
        C --> D[Entrega a UX/UI y Frontend]

4.- Flujo de trabajo del equipo UX / UI.

    Carpetas utilizadas.
        /docs/UX - UI
        /design

    Actividades principales.
    -   Transformar el branding en prototipos interactivos.
    -   Definir el flujo visual y navegación de la aplicación.
    -   Producir wireframes y mockups finales.
    -   Entregar especificaciones funcionales al frontend

    Diagrama - Proceso UX / UI
        A[Wireframes Iniciales] --> B[Prototipos Interactivos]
        B --> C[Testeo UX]
        C --> D[Entrega a Frontend]

5.- Flujo de trabajo del equipo de Data

    Carpetas utilizadas:

        /data/raw
        /data/interim
        /data/processed

        /notebooks
        /src/data
        /src/models

    Proceso de trabajo:
    
    1.- Datos crudos:
        Se almacenan exclusivamente en `/data/raw`
    
    2.- Procesamiento preliminar:
        Generación de datos intermedios: `/data/interim`

    3.- Dataset final:
        Archivos limpios y listos para modelado: `/data/processed`

    4.- Exploración y pruebas:
        Notebooks ubicados en `/noteooks`

    5.- Migración a código productivo:
        Scripts en `/src/data`

    6.- Entrenamiento del modelo:
        Implementado en `/src/models/train.py`
    
    7.- Modelo final:
        Exportado como `model.pkl`

    8.- Script de predicción:
        Implemenado en `/src/models/predict.py`
    
    Diagrama - Flujo de Datos (ETL + Modelado)
        A[raw] --> B[interim]
        B --> C[processed]
        C --> D[EDA y Notebooks]
        D --> E[Scripts Productivos]
        E --> F[Entrenamiento del Modelo]
        F --> G[model.pkl]
        G --> H[Entrega a Backend]

6.- Flujo de Trabajo del Equipo Backend.
    
    Carpetas utilizadas.
        /src/backend
        /src/backend/routes
        /src/backend/services
        /src/backend/tests
        /src/backend/utils
        /src/models (solo lectura del modelo)

    Actividades Principales:
    -   Integrar el modelo exportado por Data Science (`model.pkl`)
    -   Implementar los endoints para predicción.
    -   Gestionar la lógica de negocio asociada al detector de fraudes.
    -   Desarrollar pruebas unitarias que aseguren correcto funcionamiento.
    -   Documentar la API en `/docs/api-docs`

    Diagrama - Flujo Backend.
        A[Recepción del Modelo] --> B[Implementación de API]
        B --> C[Servicios y Utils]
        C --> D[Pruebas Unitarias]
        D --> E[Entrega a Frontend]

7.- Flujo de trabajo del Equipo Frontend

    Carpetas Utilizadas:
        /src/frontend
            ├─ api/
            ├─ components/
            ├─ hooks/
            ├─ pages/
            └─ styles/
    
    Actividades principales:
    -   Implementar la interfaz siguiento los prototipos UX / UI
    -   Consumir la API del backend para obtener predicciones.
    -   Construir una interfaz responsive y accesible.
    -   Integrar assets proporcionados por Diseño.

    Diagrama - Flujo Frontend.
        A[Recepción de Prototipos UX/UI] --> B[Construcción de Componentes]
        B --> C[Integración con API]
        C --> D[Testeo Visual y Funcional]
        D --> E[Versión Final para Integración]

8.- Reglas de Trabajo y Control de Versiones

    Reglas Generales.
    -   Ningún equipo debe modificar archivos ajenos sin autorizacion.
    -   Las carpetas de trabajo están delimitadas por rol.
    -   Los datos crudos nunca deben alterarse.

    Estrategia de Ramas

        feature/<equipo>/<función>
        bugfix/<equipo>/<corrección>
        hotfix/<urgencia>

    
    Flujo de Versionamiento.

        A[Developer] --> B[Branch Feature]
        B --> C[Commit + Push]
        C --> D[Pull Request]
        D --> E[Code Review]
        E --> F[Merge a main]

    Buenas prácticas:

    -   Mensajes de commit claros y significativos.
    -   Hacer pull antes de trabajar.
    -   Revisiones cruzadas entre equipo cuando sea necesario.

9.- Integración Final del Proyecto.

    Actividades.
    -   Pruebas end-to-end entre frontend y backend.
    -   Validación del modelo con datos reales.
    -   Revisión visual con UX /UI y diseño.
    -   Ajustes finales y reporte de observaciones.

    Diagrama - Integración final.

        A[Model + API] --> C[Frontend]
        B[Diseño + UX] --> C
        C[App Integrada] --> D[Pruebas Finales]

10.- Coordinación y comunicación entre Equipos.

    Cada equipo debe registrar en /docs/proyect-planning:
    -   Avances recientes.
    -   Requerimientos y dependencias.
    -   Bloqueos y riesgos.
    -   Próximas tareas.

    Notas importantes.
    -   La comunicación entre equipos debe ser continua.
    -   Los cambios estructurales deben documentarse inmediatamente.
